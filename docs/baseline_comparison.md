# Baseline Comparison Results

This document tracks the comparison between single-agent and multi-agent (ResQ) incident response performance.

## Methodology

The same incident (identical logs + metrics) is processed through three pipelines and scored on identical metrics:

1. **Single-Agent Baseline**: One Qwen model receives all logs + metrics in a single prompt and produces the entire response (hypotheses, root cause, remediation, post-mortem) in one pass.
2. **Multi-Agent (naive arbitration)**: The five specialized agents (Log Analyzer, Metric Monitor, Coordinator, Runbook Executor, Post-Mortem Writer) run, and the Coordinator's `ConsensusEngine` picks the highest-scoring hypothesis — but the agents never talk to each other.
3. **Multi-Agent (negotiated)**: Same swarm, plus a **negotiation round**: when the Log Analyzer and Metric Monitor disagree, each is shown the other's hypotheses and evidence and re-examines the incident (root cause vs. downstream symptom, signal timeline) before the Coordinator arbitrates. The inter-agent exchange is routed through the `MessageBus` and recorded for audit.

The three-way design isolates the value of *collaboration itself*: (2) vs (1) measures specialization alone; (3) vs (2) measures the added value of dialogue and negotiation.

## Metrics Tracked

| Metric | Description |
|--------|-------------|
| **Time to Diagnosis** | Time from incident trigger to root cause identification |
| **Diagnostic Accuracy** | Whether the correct root cause was identified |
| **Hypotheses Generated** | Number of distinct diagnostic hypotheses |
| **Evidence Quality** | Number of grounded (evidence-backed) claims vs. speculation |
| **Post-Mortem Completeness** | Coverage of timeline, root cause, impact, action items |

## Results

*Generated by `python main.py --baseline-comparison <incident.json>`. Each incident runs the identical logs+metrics through a one-call single-agent baseline and the ResQ multi-agent swarm, scored on identical metrics.*

### INC-2024-001 — ground truth: Database connection pool exhaustion

*Agents initially disagreed on the root cause: **False**. Diagnostic accuracy: naive arbitration **1.0** → after negotiation **1.0** (unchanged).*

| Metric | Single-Agent | Multi (naive) | Multi (negotiated) | Δ (negot − single) |
|--------|:------------:|:-------------:|:------------------:|:------------------:|
| Diagnostic Accuracy (0–1) | 0.5 | 1.0 | 1.0 | 0.5 |
| Hypotheses Generated | 3 | 8 | 8 | 5 |
| Evidence Quality (grounded claims) | 10 | 29 | 29 | 19 |
| Post-Mortem Completeness (0–1) | 1.0 | 1.0 | 1.0 | 0.0 |
| Time to Diagnosis (s) | 22.46 | 131.78 | 168.18 | 145.72 |

- **Single-agent identified:** A newly deployed unindexed WHERE clause in the /api/users SELECT query caused a full table scan, increasing average query time from ~100ms to >15s — starving the DB connection pool, triggering thread blocking in service-api, memory bloat under sustained GC pressure, and cascading instance failures.
- **Multi-agent (naive) identified:** Database query performance degradation causing connection pool exhaustion and cascading timeouts
- **Multi-agent (negotiated) identified:** Database connection pool exhaustion triggered by unbounded concurrent queries against an unoptimized database query, acting as the primary root cause — with downstream effects including thread blocking, latency explosion, and cascading timeouts

### INC-2024-003 — ground truth: Cache node failure causing a cache-miss storm that overwhelmed the database

*Agents initially disagreed on the root cause: **True**. Diagnostic accuracy: naive arbitration **0.0** → after negotiation **1.0** (resolved correctly).*

| Metric | Single-Agent | Multi (naive) | Multi (negotiated) | Δ (negot − single) |
|--------|:------------:|:-------------:|:------------------:|:------------------:|
| Diagnostic Accuracy (0–1) | 1.0 | 0.0 | 1.0 | 0.0 |
| Hypotheses Generated | 3 | 7 | 7 | 4 |
| Evidence Quality (grounded claims) | 12 | 25 | 26 | 14 |
| Post-Mortem Completeness (0–1) | 1.0 | 1.0 | 1.0 | 0.0 |
| Time to Diagnosis (s) | 24.03 | 131.9 | 143.53 | 119.5 |

- **Single-agent identified:** Failure of cache-node-2 at 03:09:40Z caused catastrophic cache miss storm (hit rate collapsed from ~95% to <5%), which flooded the API with uncached product queries; the unindexed 'category' filter in the products query then saturated the DB connection pool and induced severe memory pressure due to buffered requests and GC thrashing — exposing insufficient JVM heap sizing and lack of circuit-breaking or graceful degradation for cache failures.
- **Multi-agent (naive) identified:** JVM heap exhaustion due to memory leak or insufficient heap allocation, triggering cascading failures
- **Multi-agent (negotiated) identified:** Cache-node-2 failure causing catastrophic cache miss surge, which overloaded the database and triggered cascading resource exhaustion (connection pool saturation, memory pressure, GC thrashing)


## Expected Outcomes

- **More hypotheses & higher evidence quality**: parallel specialization surfaces more evidence-grounded claims than a single perspective.
- **Robustness on ambiguous incidents**: a single agent can anchor on a dramatic-but-misleading signal; the negotiated swarm cross-references specialists to distinguish root cause from symptom.
- **Conflict resolution via dialogue**: the negotiation round lets a well-supported minority hypothesis override a loud-but-wrong one — something naive arbitration cannot do.
- **Cost trade-off**: the swarm makes many more model calls, so it trades latency for diagnostic robustness. The comparison reports this honestly (Time to Diagnosis).
